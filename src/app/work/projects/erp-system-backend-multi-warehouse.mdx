## Overview

A distribution business operating across multiple warehouses was running entirely on fragmented manual workflows — spreadsheets, disconnected tools, and paper-based processes that created reconciliation overhead, inventory inaccuracies, and reporting delays. This project built a centralised ERP platform from the ground up to replace those workflows with a single, integrated system covering inventory, sales, procurement, point-of-sale, financial ledger, payroll, and daily operational cost tracking.

The system serves multiple roles — Administrator, Warehouse Head, and Warehouse Admin — with data visibility scoped at the route level to each warehouse and role.

## Problem

The business had no single source of truth for inventory, sales, or finance. Stock counts were manual and error-prone. Sales orders were tracked in spreadsheets. Financial reconciliation happened manually at month-end, consuming days of staff time. There was no visibility into per-warehouse margins, no audit trail for stock movements, and no systematic way to track outstanding customer debt.

## My Role

Designed and built the core web application — backend API architecture in Node.js and Express, database design in PostgreSQL with Sequelize, and all 25+ functional modules from inventory and sales through to financial ledger and payroll. Also responsible for user onboarding, operational training, and aligning system behavior with real-world business workflows directly validated with the operations team during development.

## Key Challenges

**Replacing deeply entrenched manual workflows.** Users had built operational habits around spreadsheets and paper processes over years. The system had to match how the business actually operated — not impose a different model — or adoption would fail regardless of technical quality.

**Complex multi-module transactional integrity.** Creating a Sales Order requires simultaneously creating line items, deducting stock from the correct warehouse rack, writing an immutable adjustment history entry, and updating customer debt — all as a single atomic operation. Any partial failure produces inconsistent state across financial and inventory records.

**High-volume reporting under operational load.** The business generates Excel reports across every operational document type — delivery orders, stock opname, purchase orders, financial statements — while the system is actively in use. Report generation had to run without degrading response times for transactional operations.

## Technical Highlights

- **Atomic transactions across service boundaries.** All multi-table write operations run inside Sequelize managed transactions. Services accept an external transaction parameter (`t = null`) to support chaining across service boundaries — a deliberate pattern for nested transactional integrity across 25+ modules
- **Built an internal printer service architecture** enabling remote cashier printing through API-triggered local print servers — eliminating the need for direct device access from the web application
- **Unique document code generation with collision prevention.** A shared `codeGenerator` utility generates prefixed codes (e.g. `SO-12345678`, `POS-87654321`) using a `do-while` loop that re-generates on collision, guaranteeing uniqueness without database sequence dependencies
- **Reusable query builder.** `buildQueryOptions` and `buildPaginationResponse` centralise pagination, filtering, search, and date-range logic across 25+ service files — eliminating repetitive query construction and keeping services focused on domain logic
- **DB-backed caching layer** with configurable TTL (default 3 hours) via a `getOrSetCache` helper — avoiding a Redis infrastructure dependency while keeping dashboard query performance acceptable at operational scale
- **Excel report generation** via ExcelJS with dynamic transaction logic, custom formatting, and multi-sheet structures across every operational document type
- Delivered **user onboarding and operational training** before full system migration — ensuring staff could operate the system confidently before manual workflows were retired

## Engineering Decisions

**Role-scoped data isolation without a multi-tenancy framework.** Notification counts, stock queries, and sales data are dynamically filtered by `user.roleId` and `user.warehouseId` at the query level — enforcing data boundaries with standard Sequelize where clauses rather than a schema-per-tenant or row-level security approach. Simple, maintainable, and sufficient for the operational scale.

**Paranoid soft-delete with actor tracking.** Critical tables use Sequelize's paranoid delete, preserving deleted records while capturing `deletedBy` and `restoredBy` references. This gave the operations team confidence that accidental deletions could be recovered, and provided a full audit trail for compliance.

**Controller → Service → Model layering enforced consistently.** All 25+ modules follow the same architecture: route → controller (request/response) → service (business logic) → model (data access). No business logic in controllers, no queries in services. This made the codebase predictable and allowed new modules to be added quickly without re-learning the structure.

## Outcome

The system replaced fragmented manual operations across procurement, inventory, sales, POS, finance, and payroll in a single platform. Inventory accuracy improved immediately — stock discrepancies that previously required manual audits are now captured automatically by the adjustment history. Monthly financial reconciliation dropped from days to hours. The operations team has full visibility into per-warehouse activity, outstanding customer debt, and daily cost tracking in real time.
