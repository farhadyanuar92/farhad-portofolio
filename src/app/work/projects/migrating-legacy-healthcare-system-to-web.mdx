## Overview

Kairos is a large-scale financial ledger system supporting hospital operations across more than 30 hospitals. The platform handles inpatient and outpatient billing, admission workflows, transaction processing, and financial reporting — all workflows that had previously run on a legacy Windows desktop application accumulated over years of production use.

The project was not a greenfield build. It was a high-stakes migration of a live system, where the web application had to run in parallel with the desktop system until a full cutover was possible. Every transaction processed had a direct financial consequence for the hospital's operational revenue.

## Problem

The existing desktop system was tightly coupled, undocumented, and inaccessible outside the local hospital network. Business logic was embedded across client-side code without separation, and there was no path to multi-hospital support without fundamental rearchitecting. The organisation needed a web-based platform that could serve multiple hospitals simultaneously while preserving years of existing financial logic and maintaining transaction accuracy.

## My Role

Full-stack engineer responsible for backend API development in .NET Core and C#, frontend implementation in Next.js and TypeScript, financial transaction processing, integration with existing hospital infrastructure, and analysing legacy desktop code to understand and correctly reimplement undocumented business logic.

## Key Challenges

**Reverse-engineering undocumented business logic.** The desktop system had no documentation. Every financial rule — charge structures, discount logic, admission state transitions, billing calculations — had to be extracted by reading production behavior, querying key operational users, and cross-referencing output against the existing system.

**Parallel system operation.** During the transition period, both systems operated simultaneously against shared infrastructure. Data consistency between the desktop and web systems was non-negotiable. Any divergence in financial records was a production incident.

**Financial accuracy under time pressure.** In healthcare finance, incorrect calculations affect patient invoices and hospital revenue. The engineering bar for correctness was unusually high — there was no tolerance for rounding errors, missing charges, or incorrect state transitions.

## Technical Highlights

- Implemented parallel processing strategies for large-scale ledger operations handling **100,000+ transactional records** while maintaining system stability and financial accuracy
- Rebuilt the backend in **.NET Core (C#)** with **xUnit** unit tests maintaining **90%+ coverage** throughout — every service and domain rule covered before any deployment to the shared environment
- Wired all code into **SonarQube** for continuous static analysis, with every pull request gated on bugs, code smells, and coverage checks before merge
- Rebuilt the frontend in **Next.js and TypeScript** with **Jest** component-level unit tests and SonarQube coverage gates enforced consistently
- Implemented a thin adapter layer over the existing hospital database, allowing the web frontend to be delivered independently of the full backend rewrite timeline

## Engineering Decisions

**Strangler fig migration over big-bang rewrite.** New modules replaced legacy components incrementally — admission, billing, then reporting — with each module validated against the desktop system before the next phase began. This avoided a risky full cutover and gave operational teams time to verify correctness in each area.

**Test coverage as a hard constraint.** Given the financial stakes, 90%+ xUnit coverage was treated as a non-negotiable baseline, not a target. SonarQube gates on every PR enforced this throughout the project lifecycle, not just at launch.

**Adapter pattern over direct migration.** Rather than migrating the database schema immediately, a thin .NET API was placed over the existing schema, allowing the frontend to be built and deployed while the data layer migration happened in parallel.

## Outcome

Clinical and finance staff gained access to the system from any device without installing software. The engineering team inherited a codebase with real test coverage and automated quality analysis. The staged migration delivered the transition without a single disruption to patient-facing financial operations. The platform now supports active usage across more than 30 hospitals.
